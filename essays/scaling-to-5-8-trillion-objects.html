Scaling to 5.8 Trillion Objects: Lessons from Building Meta's Data Classification Engine

Mechanisms of Automatic Data Classification

Automatic data classification is the foundational capability that allows organizations to securely manage, organize, and categorize their data assets based on predefined sensitivity criteria, such as the presence of Personally Identifiable Information (PII) or Protected Health Information (PHI). To handle massive volumes of both structured and unstructured information, modern classification systems replace manual tagging with programmatic logic and sophisticated artificial intelligence. The core objective of these architectures is to map an organization's digital footprint, identifying exactly where sensitive data resides across highly distributed, constantly evolving storage environments without imposing constraints on the underlying data format.

The operational mechanics of an automated classification pipeline typically involve four distinct phases. Initially, the system catalogs metadata from disparate data stores to build a lightweight index of available assets, which prevents downstream processing from overwhelming client databases. The system then estimates the required computational resources to scan these assets, dynamically triggering localized scanning jobs. During the scanning phase, data is sampled—often using statistical techniques like Bernoulli sampling—and broken down into column-level features. Finally, a prediction service evaluates these features to assign probability-based classification labels.

To achieve high accuracy across diverse data modalities, systems deploy a hybrid array of detection techniques:

Classification Technique Underlying Mechanism Primary Application and Strengths
Rule-Based Heuristics Deterministic logic, regular expressions (Regex), and keyword extraction. Highly effective for structured data with predictable formats, such as Social Security Numbers or credit card sequences.
Natural Language Processing (NLP) Deep learning embeddings, continuous Skip-gram models, and Named Entity Recognition (NER). Ideal for unstructured text and user-generated content, providing semantic context and reducing false positives.
Decision Trees Algorithms that deduce document attributes by systematically answering sequential questions. Useful for narrowing down distinct categories by logical elimination of features.
K-Nearest Neighbors (KNN) Comparing new, unclassified data points against similar, previously labeled data points in a multidimensional space. Effective for predicting document categories and identifying clusters of highly related information.
Neural Networks Architectures processing text-based sparse features and numeric dense features through activation layers. Capable of exceeding human performance in complex language processing and intent classification tasks.

Strategic Best Practices for Automated PII Detection

The implementation of automated PII detection must be governed by rigorous regulatory frameworks and continuous algorithmic refinement. Foundational standards, specifically NIST Special Publication 800-122 and ISO/IEC 27001:2022, define the architectural and administrative baseline for protecting personal information.

NIST SP 800-122 mandates a principle of minimization, urging organizations to collect and retain only the PII absolutely necessary for business operations. A core best practice derived from this standard is categorizing all detected PII by its "confidentiality impact level," which ensures that the strictness of access controls and encryption scales proportionally with the sensitivity of the data. Correspondingly, ISO 27001 establishes specific control mechanisms for information security. Annex A Control 5.12 strictly cautions against both the under-classification and over-classification of information, requiring a balanced scheme that reflects actual business impact. Furthermore, ISO 27001 Annex A 5.34 requires organizations to establish distinct, localized policies dedicated exclusively to PII protection, accounting for varying international data privacy laws. Modern systems enforce these policies by integrating controls like data masking (Annex A 8.11) and automated data leakage prevention (Annex A 8.12) directly into the classification pipeline.

Beyond governance, best practices dictate the use of active metadata platforms and Human-in-the-Loop (HITL) workflows. Active metadata ensures that once a system detects PII, it automatically applies tags that propagate through data lineage pipelines, effectively forcing downstream analytical tables to inherit the security protocols of their upstream sources.

Because purely algorithmic systems will encounter edge cases, routing low-confidence predictions to a human review queue is essential for maintaining system integrity. To improve the baseline accuracy of these detection models, leading organizations employ robust data augmentation strategies. For example, to combat "cold start" issues when identifying novel semantic types, practitioners synthesize training data by intentionally varying sentence structures and omitting contextual clues, sometimes utilizing a 100:1 ratio of synthetic labels to actual labels. To reduce false positive rates when detecting malicious or ambiguous payloads, systems generate pseudo-random strings—including valid URIs and complex XML content—as negative samples. This negative sampling desensitizes the neural networks to irrelevant keywords, forcing the classifier to evaluate higher-level structural and semantic meanings rather than triggering erroneously on isolated tokens.

How We Built the Data Classification System at Meta

At Meta, we faced a formidable challenge: mapping and classifying sensitive semantic types across an exponentially growing, constantly shifting ecosystem of data assets. Traditional Data Loss Prevention (DLP) systems approach data security by fingerprinting specific assets and monitoring endpoints. Given the sheer scale of our infrastructure—containing trillions of data assets—this legacy approach was completely non-scalable and entirely ineffective for discovering what specific data resided in which data store. We recognized the need for an end-to-end privacy system capable of detecting sensitive semantic types at scale to automatically enforce data retention and access controls.

To solve this, we designed an automated classification infrastructure that relies on a highly scalable cloud-based architecture, multiple data signals, and a sophisticated combination of machine learning and heuristic rules. This system is deployed across a cluster of over 2,000 distributed classification instances, processing over 1 PB of daily ingested data and successfully classifying 5.8 trillion objects daily.

In designing a system of this magnitude, we had to address three core architectural bottlenecks:

  • I/O Overhead: Fetching trillions of objects from distributed filesystems (like HDFS or Warm Storage) saturates network links. To solve this, we pushed the classification logic as close to the storage layer as possible, running lightweight daemons directly on storage nodes to perform initial filtering and metadata extraction.
  • Compute Density: Running complex natural language processing (NLP) and computer vision models on CPU-only instances is prohibitively expensive. We implemented dynamic model tiering, using fast, low-power linear classifiers or heuristic models to evaluate the object first, reserving expensive GPU-based transformer models for the remaining 8% of ambiguous or high-risk objects.
  • Label Volatility: Privacy policies and classification taxonomies change frequently, requiring backfills and re-scans of historical data. We decoupled the ingestion, tokenization, and inference stages using a highly optimized, lock-free ring buffer, allowing us to backfill historical data asynchronously without impacting real-time classification throughput.

For our persisted, offline data stores, the process begins with cataloging. We collect basic metadata from every offline data store to build a comprehensive asset catalog, which ensures we can retrieve data efficiently without overloading developer resources or client databases. Using this catalog, the system estimates the required compute resources and invokes scanning jobs. Rather than scanning every single row—which would be computationally prohibitive—each compiled binary job performs a Bernoulli sample on the latest available data. We break these samples down into individual columns, extract features, and feed them into a prediction service that predicts privacy labels for each column based on ML algorithms and static rules.

Simultaneously, we must protect non-persisted, in-transit traffic. We built a live, online API that generates real-time classification predictions for data flowing across our network. This API is heavily utilized to classify egress traffic, evaluate data streaming into other machine learning models, and secure any real-time ingress streams.

Early on, we realized that relying strictly on manual heuristic rules yielded subpar accuracy, particularly when classifying highly unstructured data like user-generated content. To overcome this, we built a dedicated PyTorch-based machine learning component. This modularized PyTorch model handles over 100 data types and separates feature processing: it defines distinct embeddings to handle text-based sparse features and numeric-based dense features. These processed features form a vector that we pass through non-linear activations to output a confidence score between 0 and 1 for each label. A critical aspect of this model is our use of batch normalization; by making normalization a core part of the architecture for each training mini-batch, we achieved the same accuracy with vastly fewer training steps while learning more regular word representations. This modular design allowed us to tune specific components independently without degrading overall performance, allowing the production system to achieve average F2 scores of 0.9+ across various privacy classes.

As Meta transitioned deeper into the AI-native era, the complexity of asset classification grew. An identically named field, such as "age," might be highly sensitive personal data in a user profile, but merely a cache time-to-live (TTL) value in system metadata. To handle this deep contextual ambiguity without relying on pure manual review, we evolved our architecture into what we call Privacy-Aware Infrastructure (PAI).

Under the PAI hybrid operating pattern, we learned that rich context beats prompt engineering. Instead of passing noisy fields directly to an LLM, we gather signals from a "context mesh" that includes source code, data lineage, code searches, and ownership. We compile these signals into a structured "evidence brief." Crucially, we mask any pre-existing privacy labels in this brief to prevent the model from grading its own homework. We also decouple evaluation from optimization. We use a "multi-panel judge" framework where three independent LLMs evaluate the data using different strategies—direct-from-evidence, critique-then-classify, and metadata-only—before finalizing a highly calibrated confidence score. Finally, we distill stable model behavior into deterministic rules. While LLMs handle novel and highly ambiguous assets (roughly 15% of our traffic), once a pattern is validated, we export it as an immutable, low-latency rule in Python, SQL, or Hack. This allows us to process 85% of our classification requests with zero LLM dependency in single-digit milliseconds, keeping our infrastructure secure, auditable, and exceptionally fast at global scale.
